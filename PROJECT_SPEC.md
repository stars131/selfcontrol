# SelfControl 项目规格

## 1. 项目定位

SelfControl 是一个面向长期个人使用、同时预留多人协作与共享扩展能力的多模态备忘录与个人知识系统。

它不是简单的“记事本”或“待办清单”，而是一个可持续积累、可搜索、可地图回看、可由 AI 助手参与整理与检索的个人内容空间。

系统当前重点服务以下需求：

- 记录吃到的好吃或不好吃的餐饮、零食、饮品
- 记录日常经验、踩雷事项、备忘信息
- 上传语音、图片、视频并自动提取可搜索文本
- 通过聊天方式新增内容、查询内容和回顾内容
- 结合位置、时间线和结构化卡片进行复盘

## 2. 用户与场景

### 2.1 当前用户

- 当前以单用户自用为主
- 注册方式先采用账号密码
- 部署环境以 Linux 服务器为主
- 开发环境可以是 Windows 主机 + Linux 虚拟机

### 2.2 未来扩展

- 多用户工作区
- 工作区共享与权限分级
- 分享部分库内容或分享链接
- 面向中国大陆优先，同时保留海外地图与 AI provider 扩展能力

### 2.3 核心场景

- 记录某天吃到的好吃饭店、零食、饮品，并在之后快速查回
- 记录不好吃或不满意的体验，避免下次忘记
- 上传语音、图片、视频，让系统自动识别内容并沉淀到知识库
- 问助手“我上次在某地吃过什么”“有哪些不推荐的零食”
- 在地图上回看自己去过的地点与相关记录
- 给记录设置提醒，例如下次去某商场时不要重复购买某个零食

## 3. 产品形态

### 3.1 交互形态

- 以聊天界面为主入口
- 主界面采用“左侧聊天，右侧结构化结果卡片”模式
- 同时保留结构化录入、列表筛选、时间线和地图复盘能力
- 提供一个尽量简洁的快速录入入口，支持一条输入直接创建内容

### 3.2 终端要求

- 桌面端优先支持复杂整理、地图查看、媒体预览、设置管理
- 手机端优先支持快速录入、聊天查询、语音上传、拍照上传
- 使用响应式 Web 作为统一形态
- 首页支持语言切换

### 3.3 设置与配置

- 大模型和 AI provider 配置集中放在设置页
- 工作区主页面不重复保留 provider 配置入口
- 每项 AI 能力可以独立开启、关闭和切换 provider

## 4. 核心功能范围

### 4.1 通用记录系统

记录对象采用统一 `Record` 模型，保证长期可扩展。

当前记录应支持：

- 标题
- 正文内容
- 类型 `type_code`
- 评分
- 是否避雷 `is_avoid`
- 发生时间
- 地点信息
- 附件列表
- 来源方式
- 扩展元数据

记录对象必须支持未来继续增加更多业务类型，而不需要推翻现有模型。

### 4.2 多模态输入

系统支持：

- 文本录入
- 语音上传或录音
- 图片上传
- 视频上传
- 聊天助手解析自然语言并转成结构化记录

所有媒体在识别后都要把有效文本沉淀下来，便于后续搜索和 RAG 召回。

### 4.3 搜索与知识库

搜索体系采用混合模式：

- 结构化筛选
- 关键词搜索
- RAG 向量检索

知识库要求：

- 记录正文可切片入库
- OCR / ASR / 视频转写结果可切片入库
- 记录更新后可自动重建相关知识片段
- 聊天助手可调用知识库回答问题

### 4.4 地图与位置

地图能力当前优先基于高德地图。

要求：

- 位置先自动识别
- 支持手动调整和纠正
- 每条记录可以展示地点
- 支持位置筛选与地图钻取
- 支持地点回看，类似相册按地点查看
- 保留位置修正历史

国际化要求：

- 当前优先支持中国大陆地图能力
- 架构上预留 Google Maps、Mapbox 或自定义 provider 的替换空间

### 4.5 提醒

提醒当前采用 V1 方案：

- 以站内提醒为主
- 可与记录绑定
- 支持到期扫描生成通知
- 未来再扩展更复杂的跨端提醒或系统级推送

### 4.6 聊天助手

聊天助手是核心入口之一，当前和后续应覆盖：

- 新增记录
- 查询记录
- 返回结构化结果卡片
- 调用知识库做回答
- 创建提醒
- 协助分类、总结和回顾

后续可以继续增强：

- 更丰富的工具调用
- 更细的多轮确认
- 更强的总结、周报、月报与复盘能力

### 4.7 分享与共享

系统区分两类能力：

- 共享空间：多成员共同维护一个工作区
- 分享链接：把工作区访问权限安全地分享给其他账号

要求：

- 先做安全的工作区分享
- 支持权限级别
- 支持禁用、过期、次数限制
- 为未来更细粒度的内容分享保留空间

### 4.8 工作区迁移与大体积数据

考虑到图片、音频、视频会不断增长，系统需要：

- 媒体存储统计
- 保留策略
- 归档能力
- 导出与导入能力
- 对本地和远端存储统一处理

## 5. AI Provider 策略

AI 能力不能绑定单一供应商，必须按功能拆分配置。

当前按 feature 维度拆分：

- `chat_generation`
- `embeddings`
- `image_ocr`
- `audio_asr`
- `video_transcription`
- `maps_geocoding`
- `media_storage`

每项能力都应支持：

- 单独启用或关闭
- 独立 provider
- 独立模型名
- 独立 API 地址
- 独立 Secret 环境变量
- 健康状态与告警提示

## 6. 安全与权限要求

即使当前主要只有一个人使用，也必须从一开始按未来多人场景设计。

最低要求：

- 服务端强制权限校验
- 工作区边界隔离
- `owner / editor / viewer` 角色控制
- 关键写操作保留审计日志
- 不在数据库保存第三方明文密钥
- 上传、下载、删除媒体必须经过鉴权
- 生产环境拒绝弱密钥和不安全默认配置

## 7. 工程与架构要求

### 7.1 技术栈

- 前端：Next.js
- 后端：FastAPI
- 数据库：PostgreSQL
- 向量检索：RAG 基础能力
- 队列与异步：Redis + Celery
- 部署入口：Docker Compose + Nginx

### 7.2 架构原则

- 前端、API、后台任务、媒体存储、provider 适配层保持清晰边界
- 优先做稳，再持续增强
- 每次改动都要考虑可维护性和可更新性
- 按大型企业级项目标准推进，而不是一次性脚本式开发
- 部署、回滚、迁移、备份和恢复都要有明确路径

### 7.3 可维护性要求

- 复杂能力通过独立模块抽象，不堆在单个巨型文件里
- Provider 相关逻辑必须通过稳定适配层隔离
- 文档必须跟随工作流、部署和安全边界同步更新
- 每个有意义的切片都必须更新进度文件、验证、提交并推送

## 8. 部署要求

- 开发机可以是 Windows
- 推荐主要开发运行环境为 Linux 虚拟机
- 线上服务器为 Linux
- 支持开发环境和生产环境分离
- 支持反向代理、TLS、健康检查、备份与恢复

## 9. 当前阶段定义

当前阶段目标不是一次性做到所有高级能力，而是完成一个能长期演进的稳定底座。

阶段目标包括：

- 用户可注册、登录、进入工作区
- 可用聊天和结构化方式创建记录
- 可上传媒体并做文本提取
- 可搜索记录与知识库
- 可看时间线和地图
- 可创建提醒并查看通知
- 可配置 provider
- 可分享工作区
- 可导出导入工作区
- 可在 Linux 服务器稳定部署

## 10. 验收标准

一个阶段性交付至少应满足：

1. 主流程可用，不依赖手工改库才能工作。
2. 关键安全边界正确，权限绕不过去。
3. 多媒体内容可进入可搜索流程。
4. 地图、时间线、聊天、结构化结果能形成闭环。
5. 提醒、分享、导入导出具备基本可用性。
6. 文档、验证脚本、部署脚本和进度文件保持同步。
7. 改动可以被持续维护、持续更新，而不是只能一次性运行。
