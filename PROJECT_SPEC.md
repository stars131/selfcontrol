# SelfControl 项目规格

## 1. 项目定位

SelfControl 是一个面向个人长期使用、同时保留多人协作扩展能力的多模态备忘与知识检索系统。

系统目标不是做“简单记事本”，而是做一个可长期积累、可搜索、可追溯、可由 AI 协助整理和调用的个人知识空间。

核心使用方式：

- 以聊天界面为主入口
- 左侧负责聊天、提问、录入和助手交互
- 右侧负责展示结构化结果卡片、地图、筛选结果、时间线、提醒和诊断信息

## 2. 目标用户

当前阶段：

- 单个用户自用

未来扩展：

- 多用户工作区
- 空间共享
- 记录库分享
- 角色与权限控制

## 3. 产品目标

### 3.1 核心目标

- 支持文字、语音、图片、视频等多模态内容录入
- 将媒体内容自动转为可搜索文本
- 支持结构化筛选、全文搜索和 RAG 语义检索
- 支持地图位置识别、展示、纠错和钻取
- 支持提醒、回顾和复盘
- 支持 AI 助手完成新增、查询、整理、总结等操作
- 支持多模型供应商拆分配置
- 从第一天开始按企业级可维护性和可扩展性标准设计

### 3.2 非目标

第一阶段不优先投入：

- 原生 iOS / Android App
- 历史第三方数据导入
- 复杂视频内容理解
- 浏览器外长期后台地理围栏提醒

## 4. 使用场景

典型场景包括：

- 记录某天吃到的好吃餐厅、零食、饮料
- 记录踩雷项，避免再次购买或再次去不满意的地点
- 上传语音、图片、视频，让系统自动识别内容并保存为可检索知识
- 通过自然语言提问，例如“我去年在杭州吃过哪些不错的日料”
- 在地图上回看曾经去过的地方和对应记录
- 创建提醒，例如“下次去某个商场时提醒我不要买某个零食”
- 将个人库中的某些内容安全分享给其他人

## 5. 终端形态

- 前端形态：响应式 Web
- 桌面端：优先支持整理、筛选、地图回看、媒体预览、配置管理
- 手机端：优先支持快速录入、语音录入、拍照上传、聊天查询
- 开发环境：Windows + Linux 虚拟机
- 服务器部署：Linux

## 6. 核心功能范围

### 6.1 记录系统

记录对象采用通用 `Record` 模型，支持后续持续扩展。

首批记录类型：

- `food`
- `snack`
- `drink`
- `bad_experience`
- `memo`

后续可扩展：

- `travel`
- `shopping`
- `movie`
- `book`
- `health`

每条记录应支持：

- 标题
- 正文描述
- 类型
- 分类
- 标签
- 评分
- 是否避雷
- 发生时间
- 地点信息
- 附件列表
- 来源方式
- 扩展 JSON 字段

### 6.2 多模态录入

支持：

- 文字录入
- 语音上传或录音
- 图片上传
- 视频上传
- 聊天助手解析自然语言并生成记录草稿

### 6.3 媒体处理

上传后的媒体进入异步处理链路：

- 音频：转写、摘要、关键词提取
- 图片：OCR、图像描述、EXIF 信息提取
- 视频：音轨提取、转写、摘要、封面和基础元数据解析

处理结果必须：

- 可搜索
- 可挂到对应记录
- 可进入知识库
- 可供聊天助手引用

### 6.4 搜索系统

搜索采用三层混合模式：

- 结构化查询：类型、分类、标签、时间、地点、评分、避雷等
- 全文搜索：正文、转写文本、OCR 文本、摘要
- RAG 检索：向量召回 + 可选重排

### 6.5 地图系统

地图能力基于高德地图，后续预留海外地图替换能力。

要求：

- 自动识别地点
- 支持手动修正地点
- 地图点位展示
- 地点筛选
- 地图钻取
- 按地点回看记录
- 保留位置修正历史

地点来源优先级：

1. 媒体 EXIF / 元数据
2. 文本内容推断
3. 地理编码服务
4. 用户手动修正

### 6.6 提醒系统

提醒采用 V1 方案，优先实现站内提醒。

支持：

- 时间提醒
- 周期提醒
- 条件提醒
- 基于当前位置的站内提示能力预留

### 6.7 聊天助手

聊天助手支持：

- 新增记录
- 查询记录
- 修改记录
- 删除记录
- 归类和打标签
- 创建提醒
- 生成摘要、周报、月报
- 调用知识库回答问题
- 根据既有记录提出整理建议

写入类操作流程：

- 助手解析用户意图
- 生成结构化草稿
- 用户确认后写入

### 6.8 分享与共享

系统区分两类能力：

- 共享空间：多人共同维护同一个工作区
- 记录库分享：将某个空间或部分内容分享给他人

要求：

- 安全的分享链接
- 可控权限
- 可禁用、可限次
- 后续支持指定用户共享

### 6.9 AI Provider 配置

各类 AI 能力必须可拆分配置，不能绑定单一供应商。

按能力拆分：

- 聊天模型
- Embedding 模型
- OCR 模型
- 语音转写模型
- 图像理解模型
- 视频转写 / 理解模型
- 媒体存储提供方

每项配置应支持：

- 是否启用
- 默认供应商
- 模型名
- API 地址
- API Key 环境变量名
- 超时
- 重试策略
- feature 级开关

## 7. 数据与知识库

### 7.1 数据层

核心表包括：

- `users`
- `workspaces`
- `workspace_members`
- `records`
- `categories`
- `tags`
- `record_tags`
- `media_assets`
- `places`
- `conversations`
- `messages`
- `reminders`
- `notifications`
- `shares`
- `provider_configs`
- `audit_logs`

### 7.2 知识库层

知识库基于 RAG 和向量存储，要求：

- 记录正文可切片入库
- OCR / ASR / 视频转写结果可入库
- 记录更新后自动重建相关知识索引
- 聊天检索可以引用来源记录和片段

## 8. 安全与权限

### 8.1 当前阶段

- 注册方式：账号密码
- 默认单用户
- 仍按多用户安全边界设计

### 8.2 未来阶段

- 角色区分：`owner` / `editor` / `viewer`
- 分享权限控制
- 空间成员管理
- 只读共享视图

### 8.3 基础安全要求

- API Key 不直接存数据库明文
- 环境变量方式管理敏感密钥
- 工作区数据隔离
- 审计日志记录关键操作
- 上传、下载、删除媒体必须鉴权

## 9. 架构要求

### 9.1 技术栈

- 前端：Next.js
- 后端：FastAPI
- 数据库：PostgreSQL
- 向量扩展：pgvector
- 缓存与消息队列：Redis
- 异步任务：Celery
- 部署入口：Nginx + Docker Compose

### 9.2 架构原则

- 前后端边界清晰
- 工作区维度隔离
- 媒体文件与业务元数据分离
- AI Provider 统一抽象
- 所有耗时任务可异步
- 可维护、可更新、可回滚
- 满足大型企业级工程标准

## 10. 部署要求

- 开发：Windows + Linux VM
- 线上：Linux
- 支持 Docker Compose 开发与生产部署
- 支持反向代理、TLS、日志轮转、备份恢复
- 生产环境需开启严格配置校验

## 11. 当前实施策略

当前采用“先做稳定 V1，再持续企业级打磨”的策略。

优先级：

1. 主流程可用
2. 数据安全
3. 可维护性
4. 可扩展性
5. 运维与部署稳定性

## 12. 验收标准

项目达到阶段性交付时，至少应满足：

- 可注册、登录、进入工作区
- 可通过聊天或表单创建记录
- 可上传图片、音频、视频并完成文本提取
- 可搜索记录、检索知识库、展示来源
- 可查看地图并进行地点纠正和筛选
- 可创建提醒并查看通知
- 可配置 AI Provider
- 可分享工作区或分享链接
- 可在 Linux 服务器稳定部署
- 具备基础测试、校验脚本、审计和回滚能力
