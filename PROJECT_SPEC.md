# SelfControl 项目文档

## 1. 项目定位

SelfControl 是一个面向个人、可扩展到多人协作的多模态生活记录与智能检索系统。

系统核心目标：

- 支持文字、语音、图片、视频等方式记录内容。
- 对多媒体自动进行转写、OCR、图像理解、摘要提取，形成可检索内容。
- 支持结构化筛选、全文检索、RAG 语义检索混合查询。
- 提供地图视图，支持地点自动识别、手动修正、位置回看。
- 提供聊天式 Agent，完成新增、查询、修改、删除、整理、总结、提醒创建等操作。
- 当前优先单用户使用，但底层从第一天开始支持多人、共享空间、权限控制和分享。

## 2. 使用场景

典型场景包括：

- 记录哪天吃到了好吃的饭、零食、饮料。
- 记录踩雷项，避免重复购买或重复去不满意的店。
- 上传图片或视频，让系统自动识别内容并保存文字化结果。
- 通过自然语言查询过去的记录，例如“我去年在杭州吃过哪些不错的日料”。
- 在地图上回看曾经去过的地方和相关记录。
- 创建提醒，例如“下次到这个商场时提醒我别买这款零食”。
- 未来与他人共享空间或分享自己的知识库。

## 3. 产品目标

### 3.1 核心目标

- 建立一个稳定的个人知识与记忆系统。
- 让多模态内容都可转为可搜索、可总结、可回看的知识资产。
- 通过聊天交互降低录入与检索成本。
- 保证未来可扩展到多人、共享和多地图/多模型供应商。

### 3.2 非目标

- 第一版不做原生移动端 App。
- 第一版不做复杂视频镜头级理解。
- 第一版不做历史外部数据导入。
- 第一版不追求浏览器后台实时地理围栏提醒。

## 4. 用户与角色

### 4.1 当前用户

- 单用户本人使用。

### 4.2 未来角色

- `owner`：空间所有者，拥有全部权限。
- `admin`：管理成员、配置和内容。
- `editor`：可新增、编辑、查询内容。
- `viewer`：只读访问。

## 5. 产品形态

### 5.1 交互模式

采用“左侧聊天，右侧结构化结果卡片”的工作台模式。

### 5.2 终端形态

- 响应式 Web，兼容桌面与手机浏览器。
- 桌面端偏重整理、筛选、地图回看、媒体预览。
- 手机端偏重快速新增、语音录入、拍照上传、聊天查询。

## 6. 功能范围

### 6.1 记录系统

记录对象采用通用 `Record` 模型，支持后续动态扩展。

首批记录类型：

- `food`
- `snack`
- `drink`
- `bad_experience`
- `memo`

后续可扩展：

- `travel`
- `movie`
- `shopping`
- `book`
- `health`

每条记录建议包含：

- 标题
- 正文描述
- 记录类型
- 分类
- 标签
- 评分
- 是否避雷
- 发生时间
- 地点
- 附件列表
- 来源方式
- 扩展字段 JSON

### 6.2 多模态录入

支持以下输入：

- 文字录入
- 语音上传或录音
- 图片上传
- 视频上传
- 聊天助手解析自然语言并生成记录草稿

### 6.3 多媒体处理

媒体上传后进入异步处理链路：

- 语音：转写、摘要、关键词提取
- 图片：OCR、图片描述、EXIF 时间地点提取
- 视频：音轨提取、转写、摘要、封面生成、元数据解析

处理结果保存为结构化数据和可检索文本。

### 6.4 检索系统

检索采用三层混合方案：

- 结构化查询：分类、标签、时间、地点、评分、避雷等
- 全文检索：记录正文、转写文本、OCR 文本、摘要
- RAG 检索：语义向量召回 + 可选重排序

### 6.5 地图系统

地图相关能力：

- 自动识别地点
- 手动调整地点
- 高德地图展示
- 点位查看
- 地点聚合
- 按时间或分类回看

地点来源优先级：

1. 媒体 EXIF 或视频元数据
2. 文本内容推断
3. 地理编码服务
4. 用户手动修正

用户手动修正后，作为最终地点结果保存。

### 6.6 提醒系统

提醒采用 V1 方案，优先实现站内提醒能力。

提醒类型：

- 时间提醒
- 周期提醒
- 条件提醒
- 位置提醒（在用户打开系统时，结合当前位置触发）

第一版不做浏览器外持续后台地理围栏。

### 6.7 聊天助手

聊天助手支持：

- 新增记录
- 查询记录
- 修改记录
- 删除记录
- 归类和打标签
- 创建提醒
- 生成摘要、周报、月报
- 调用知识库回答问题
- 基于记录提出整理建议

写入类操作流程：

- Agent 先解析用户意图
- 生成结构化草稿
- 用户确认后写入

查询类操作可以直接返回结果，并展示引用依据。

### 6.8 共享与分享

系统区分两种共享形式：

- 共享空间：多人协作维护同一个空间
- 库分享：把某个库或某个视图分享给其他人

分享方式首版建议：

- 指定用户分享
- 邀请码分享
- 有权限控制的链接分享

### 6.9 模型供应商配置

每种能力可独立开关并独立配置供应商。

按能力拆分配置：

- 聊天模型
- Embedding 模型
- 语音转写模型
- OCR 模型
- 图像理解模型
- 视频转写/理解模型
- 重排序模型

每项配置包含：

- 是否启用
- 默认供应商
- 备用供应商
- 模型名称
- API 地址
- API Key
- 超时
- 重试次数
- 成本统计

## 7. 系统架构

## 7.1 总体架构

- 前端：`Next.js`
- 后端：`FastAPI`
- 主数据库：`PostgreSQL`
- 向量扩展：`pgvector`
- 缓存与队列：`Redis`
- 异步任务：`Celery`
- 对象存储抽象：本地存储 + S3/MinIO 兼容
- Web 服务入口：`Nginx`
- 部署方式：`Docker Compose`

## 7.2 架构原则

- 结构化数据与向量数据分层管理
- 媒体文件与元数据分离
- 模型调用统一走 Provider 抽象层
- 地图能力统一走 Provider 抽象层
- 所有业务数据必须支持 `workspace` 维度权限隔离
- 所有耗时任务异步化

## 10. 检索与 RAG 设计

### 10.1 数据分层

- PostgreSQL 保存结构化数据
- `pgvector` 保存向量
- 大文件仅保存存储引用

### 10.2 分块策略

以下内容进入向量索引：

- 记录正文
- 语音转写文本
- OCR 文本
- 图片描述
- 视频摘要
- 用户补充评论

### 10.3 查询流程

1. 识别用户意图
2. 提取结构化条件
3. 按权限过滤空间
4. 执行数据库条件查询
5. 执行向量召回
6. 可选重排序
7. 汇总结果并返回引用来源

## 11. Agent 工作流

### 11.1 新增记录

1. 用户输入自然语言或上传媒体
2. Agent 识别意图与字段
3. 调用多媒体解析链路
4. 生成结构化记录草稿
5. 用户确认
6. 写入数据库
7. 生成向量索引
8. 可选创建提醒

### 11.2 查询记录

1. 用户输入自然语言
2. Agent 判断是结构化查询、地图查询、时间线查询还是语义查询
3. 调用检索服务
4. 在右侧展示记录列表、地图卡片、摘要卡片
5. 返回带引用的答案

### 11.3 修改与删除

- 修改：先展示拟修改字段，再确认提交
- 删除：建议软删除并支持回收站

## 13. 存储策略

### 13.1 原则

- 数据库存元数据
- 大文件走对象存储抽象
- 开发环境可使用本地磁盘
- 生产环境建议兼容 S3/MinIO

### 13.2 多媒体处理

- 上传后立即保存原文件
- 异步执行转写、OCR、摘要、封面生成
- 保存缩略图与预览图
- 保存处理状态与错误日志

## 14. 地图与坐标设计

当前地图服务优先使用高德。

必须保留多 Provider 能力，原因：

- 当前主要面向中国大陆
- 未来需要支持国外地图服务

字段设计需明确区分坐标系，例如：

- `GCJ-02`
- `WGS84`

避免未来不同地图服务切换时出现坐标偏移问题。

## 15. 提醒 V1 设计

提醒 V1 采用站内可落地方案：

- 时间提醒：定时任务扫描触发
- 周期提醒：按规则计算下次触发时间
- 条件提醒：在相关查询或页面加载时提示
- 位置提醒：用户打开系统并授权位置后，基于当前位置进行一次性判断

提醒通知方式：

- 站内通知
- 聊天窗口提示
- 提醒中心展示

## 18. Linux 虚拟机开发建议

建议使用 Linux 虚拟机作为主要运行环境。

推荐方式：

- Windows 作为主操作系统和编辑端
- Linux VM 作为主要运行与联调环境
- 使用 VS Code Remote SSH 连接 Linux VM
- 所有服务通过 Docker Compose 在 Linux 中运行

建议 Linux VM 准备：

- `Docker`
- `Docker Compose`
- `git`
- `Node.js`
- `Python 3.11+`
- `ffmpeg`
- `PostgreSQL`（如不完全容器化）
- `Redis`（如不完全容器化）

原因：

- 上线目标环境为 Linux
- 数据库、队列、媒体处理和 Nginx 在 Linux 下更接近生产环境
- 可减少路径、权限、依赖和部署兼容问题

建议按以下顺序推进：

1. 初始化仓库与基础目录

3. 建立数据库与基础表结构

5. 实现记录 CRUD

## 21. 下一步执行建议

进入虚拟机开发后，建议先做以下三件事：

1. 初始化 Git 仓库与 Monorepo 目录结构
2. 搭建 `Next.js + FastAPI + PostgreSQL + Redis` 开发骨架
3. 先把用户、空间、记录、媒体、地点五类核心表建起来

在此基础上，再接入聊天 UI、Agent 和 RAG。
